## 1. UX Audit (Prioritised)

---

### UX-001 — `is_full_count=False` is invisible to the user **[CRITICAL]**

- **Area:** Diff results — metric bar
- **Finding:** When key validation fails (duplicate keys), `is_full_count=False` and all diff counts may be inflated by Cartesian product expansion. The API serialises `is_full_count` correctly, but the UI (`index.html`) never reads or displays it. A user running against a file with duplicate keys sees counts like "+10,000 ADDED" that appear authoritative but are unreliable estimates.
- **Related:** P4-T1 (planned), KI-010 (fixed in backend, not in UI), D-003.
- **Fix:** Add a `[~ ESTIMATE]` or `[⚠ SAMPLE]` badge to the metric bar when `is_full_count === false`. Already planned as P4-T1.
- **Status after audit:** **FIXED** (see UX Fixes section below)

---

### UX-002 — Column Change Summary shows sample-based counts with no warning **[CRITICAL]**

- **Area:** Diff tab — Column Change Summary table
- **Finding:** The "Modified" and "Fmt-only" columns in the Column Change Summary are populated from `ColumnDiffStats.modified_count` and `formatting_only_count`, which come from the Python loop over `sem_sample` (≤200 rows). The denominator in `change_rate` is correctly `total_rows_f1`, but the numerator is capped at ~200 sample rows.
  - A column that changed in 45,000 rows, but where those rows happen not to appear in the 200-row display sample, will show **Modified: 0, Change: 0.00%**.
  - An analyst reading this table would conclude that the column had no changes — a false negative with no warning.
- **Related:** KI-018 (documents the gap; deferred to Phase 3 P3-T1).
- **Fix (immediate):** Add a visible notice in the Column Change Summary section header: "Column counts are from the display sample (≤200 rows); row-level totals in the summary bar are exact." Link to KI-018 note.
- **Status after audit:** **FIXED** (section header caveat label added)

---

### UX-003 — Export CSV downloads a sample, not the full diff — no indication **[CRITICAL]**

- **Area:** Results card — Export CSV button
- **Finding:** The "⬇ Export CSV" button downloads the diff CSV generated by `render_csv_diff()`, which contains `sample_diffs[:100]` — at most 100 rows. The metric bar shows "45,563 MODIFIED" while the downloaded CSV contains at most 100 rows. There is no label, tooltip, or warning anywhere in the UI indicating that the export is a sample.
  - This is a severe trust breach: the user runs a full-file comparison, sees exact counts, and exports — expecting all 45,563 rows but receiving 100.
- **Related:** KI-002 (export still sample-only; full export is Phase 3 P3-T5).
- **Fix (immediate):** Change button text to "⬇ Export CSV (sample)" and add a `title` attribute explaining the limit.
- **Status after audit:** **FIXED** (button labelled as sample)

---

### UX-004 — "Formatting-only" is unexplained inline **[HIGH]**

- **Area:** Metric bar, Diff tab pill filters
- **Finding:** The metric bar shows "449,437 FMT ONLY". The pill shows "Fmt-only". Neither has an inline explanation. An analyst unfamiliar with the product does not know whether this is an error, a warning, or a purely informational count. The Help modal explains it, but users typically don't read Help before interpreting results.
- **Fix:** Add a `title` tooltip on the FMT ONLY metric chip and on the "Fmt-only" pill: "Rows that differ in raw form (e.g. capitalisation, whitespace, numeric format) but are semantically equivalent. Not counted as modified."
- **Status after audit:** **FIXED** (tooltip added to metric chip and pill)

---

### UX-005 — Pill filter and column-level header dropdown both filter Type column **[HIGH]**

- **Area:** Diff tab — Sample Row Diffs toolbar
- **Finding:** The sample diffs table has TWO type-filtering mechanisms:
  1. The `[All][Added][Removed][Modified][Fmt-only]` pills (programmatic `setFilter` via `_applyTblFilters`)
  2. Tabulator's built-in `headerFilter: 'select'` dropdown on the "Type" column header
  Both filter the same column but use different Tabulator filter paths. They do AND-combine (header filters and programmatic filters stack in Tabulator 6), but visually they appear independent. If a user selects "Modified" via pill and also opens the column dropdown, they see two "Modified" selections with no indication they are stacking.
- **Fix (medium):** Remove `headerFilter` from the Type column in `initDiffTabulators` since the pills already cover type filtering. Keep `headerFilter: 'input'` on the Key column only (text search is useful there). Leave the global search box for cross-column search.
- **Status after audit:** **FIXED** (headerFilter removed from Type column)

---

### UX-006 — No key-used indicator in the results **[HIGH]**

- **Area:** Results — Diff tab, metric bar
- **Finding:** After auto-detection, the key column(s) are visible only inside the "Sample Row Diffs" section header (`Key (EmployeeID): ...`). It is not shown in the metric bar or anywhere prominent. A user who runs with auto-detection has no easily-visible confirmation of which key was used without scrolling into the Diff tab first.
- **Related:** P4-T3 (planned — "show key columns and how selected")
- **Fix:** Add a key indicator to the metric bar alongside the diff counts.
- **Status after audit:** **FIXED** (key displayed in metric bar)

---

### UX-007 — Validation checks show passed and failed checks identically **[HIGH]**

- **Area:** Validation tab
- **Finding:** `_serialize_validation` serialises all checks including their `passed` field. The UI `buildChecks` function groups by severity (CRITICAL/ERROR/WARNING/INFO) and renders all checks in each bucket. A `passed=True` INFO check (e.g., "Uniqueness: key column is unique") and a `passed=False` CRITICAL check both appear in the same accordion bucket with no visual distinction.
  - Analysts expect to scan for failures, not parse each check to determine pass/fail.
- **Fix (medium):** In `buildChecks`, add a visual indicator (e.g. `✓` in muted color) to passed checks, or filter them out by default with an "Include passed checks" toggle.
- **Status after audit:** Deferred to Phase 4 (small visual change but needs careful spec)

---

### UX-008 — Progress gives no sense of total work remaining **[MEDIUM]**

- **Area:** Progress card
- **Finding:** For a 344-second compare run, the progress bar shows: phase name ("Profiling"), detail ("File 1, 14 columns"), fractional counter (0/2), and elapsed time (34s). An analyst has no idea whether the run is 10% or 90% done — both "Profiling File 1" states look identical.
  - The phases are: Loading → Schema → Profiling (×2) → Key Discovery → Key Validation → Diffing → Validation → Exporting. The UI shows each in isolation with no global step counter.
- **Fix (medium):** Add a step counter: "Step 4 / 9" alongside the phase name. This requires counting total steps in `compare.py` and emitting a `step` field on each progress event.
- **Status after audit:** Deferred — requires backend change; low risk but small scope creep

---

### UX-009 — SSE disconnect error is not actionable **[MEDIUM]**

- **Area:** Error handling
- **Finding:** `sse.onerror = () => { if (S.running) { finishRun(); showErr('Connection lost. Check the server.'); } }`. This fires on any EventSource error, including normal job completion race conditions (the SSE stream ends before the client processes all events). The message is not actionable.
- **Fix:** Detect whether the job is complete via `/api/job/{job_id}` before showing the error. If the job completed, fetch the result rather than showing an error.
- **Status after audit:** Deferred — requires refactoring the SSE error handler

---

### UX-010 — null_introduced / null_resolved per column not surfaced **[MEDIUM]**

- **Area:** Diff tab — Column Change Summary
- **Finding:** `ColumnDiffStats` has `null_introduced_count` and `null_resolved_count` fields. These are serialised by `_serialize_diff` and reach the browser as `diff.column_diffs[col].null_introduced` and `.null_resolved`. The Tabulator column definitions do not include them.
  - For an analyst checking data quality, "Salary introduced 9,897 nulls" is actionable. It is currently invisible.
- **Fix:** Add two optional columns to the Column Change Summary table: "Null In" (null introduced) and "Null Out" (null resolved). Hidden by default; visible when non-zero values exist.
- **Status after audit:** Deferred — Phase 4 enhancement

---

### UX-011 — Escape key does not close the file browser modal **[MEDIUM]**

- **Area:** File browser modal
- **Finding:** The browse modal can be closed only by clicking ✕ or clicking outside. Pressing Escape has no effect. Standard browser UX expectation is that Escape closes modals.
- **Fix:** Add `document.addEventListener('keydown', e => { if (e.key==='Escape') closeBrowse(); })`.
- **Status after audit:** **FIXED** (Escape handler added)

---

### UX-012 — Error banner does not handle long messages **[MEDIUM]**

- **Area:** Error display
- **Finding:** `showErr(msg)` sets `el.textContent = '⚠ ' + msg`. The banner is a single-line `div.banner-err` with no max-height or overflow handling. A Python traceback or a very long file path would push other content down and create an unreadable wall of text.
- **Fix:** Set `max-height: 80px; overflow-y: auto` on `.banner-err` and change to `white-space: pre-wrap` so tracebacks are readable.
- **Status after audit:** **FIXED** (CSS updated)

---

### UX-013 — History "Re-run" CTA is not prominent **[LOW]**

- **Area:** History panel
- **Finding:** Clicking a history run shows summary counts and a note "Re-run comparison to see full results." but no button. The file paths are not loaded back into the inputs — the user must manually re-enter them.
- **Related:** Enhancement; the history `showHistRun` restores file paths correctly in the inputs... actually checking the code, it does NOT restore paths — it only shows a fake result. A user would need to re-type paths.
- **Fix (low priority):** In `showHistRun`, populate the file path inputs with `r.file1` and `r.file2` so the user can click Compare immediately. Add a "Re-run" button.
- **Status after audit:** Deferred

---

### UX-014 — Tabulator `renderVertical: 'basic'` is a future scalability risk **[LOW]**

- **Area:** Diff tab, Profile tab
- **Finding:** All rows are in the DOM simultaneously. At current caps (100 sample rows, ≤22 profile columns), this is fine. If `sample_diffs` is ever raised to 1000+, or if column count exceeds 50, DOM performance could degrade on low-spec machines.
- **Current cap:** `d.sample_diffs[:100]` in `_serialize_diff` limits DOM rows to 100 in the diff table.
- **Fix:** No action needed now. If sample size is raised in Phase 3, switch to `renderVertical: 'virtual'` and set a fixed `rowHeight` to enable virtual rendering (incompatible with word-wrap variable heights — would need a trade-off decision).
- **Status after audit:** Documented; no action

---

### UX-015 — FMT ONLY count is shown in the Validation tab too but means something different **[LOW]**

- **Area:** Cross-tab consistency
- **Finding:** In the Diff tab, "Fmt-only" means rows with raw-but-not-semantic changes. In the Validation tab, "Duplicate Keys" and "Mixed Types" appear under ERROR/WARNING severity but are formatting or schema issues, not row-level diffs. The mental model gap between the two tabs is not explained.
- **Fix:** Low priority; add inline section headers explaining "These checks are independent of the row-level diff counts above."
- **Status after audit:** Deferred

## 2. Scalability Risk Audit

---

### SR-001 — `renderVertical: 'basic'` — all rows in DOM **[LOW / ACCEPTABLE NOW]**

- Current sample caps: 100 diff rows, 22 profile columns. DOM size is trivially small.
- Risk materialises if: `sample_diffs` is raised to 1000+, or profile columns exceed 100.
- Mitigation: The `sample_diffs[:100]` cap in `_serialize_diff` is a hard guard. Do not raise it without switching to `renderVertical: 'virtual'`.

---

### SR-002 — Double full-file join scan **[MEDIUM — KNOWN]**

- Documented as KI-017. For 500k rows: ~344s total. For 5GB/13M rows: could exceed browser timeout or user patience.
- Mitigation: Planned as P3-T3 (merge semantic + raw joins). Acceptable until Phase 3.

---

### SR-003 — Column Change Summary counts are sample-based **[HIGH — KNOWN]**

- Documented as KI-018. Per-column `modified_count` comes from Python loop over ≤200 sample rows.
- Impact: Change rates are underestimates. Columns with changes outside the sample window show 0%.
- Mitigation: Planned as P3-T1. In the UI, the section header now shows a caveat (UX-002 fix).

---

### SR-004 — `_applyTblFilters` calls `Object.values(row)` — HTML in search **[LOW]**

- The global search function checks `Object.values(row)` which includes the `changes` field containing raw HTML strings. Searching "span" or "div" would match all rows. Not currently a problem (users don't search for HTML tags), but it's fragile.
- Mitigation: Add a `_searchText` field (plain text version of changes) for search, exclude `changes` from `Object.values` search scope.
- **Status after audit:** **FIXED** (search now uses dedicated `_searchText` field, excludes `changes` HTML field)

---

### SR-005 — Tabulator CDN — offline risk **[LOW]**

- If the CDN is unavailable (no internet connection on analyst machine), the Diff and Profile tabs show empty containers with no fallback.
- The browser console would show a load error, but the UI shows nothing.
- Mitigation: Add a CDN fallback check: detect if Tabulator loaded, show a static table fallback if not.
- **Status after audit:** Deferred — low priority given local-server deployment model

---

### SR-006 — SSE streaming — job TTL 1 hour **[LOW]**

- Jobs older than 1 hour are purged from the in-memory `jobs` dict. If a user leaves a tab open for >1 hour and then tries to fetch results, the job is gone.
- This is mitigated by history (results are saved to SQLite before purge).
- Mitigation: Acceptable. Document in Help.

## 7. Summary Assessment

**Product is trustworthy for correctness** (Phase 1 + Phase 2 complete). The engine produces correct full-file counts, correct classification, and stable exports.

**The primary UX risk before Phase 3** is the gap between what the UI implies (exact counts everywhere) and what some displays actually contain (sample-based column-level stats, sample-only CSV export). Three of the five critical fixes address this directly.

**No backend architecture changes** were made. All fixes are confined to `web/static/index.html`.

**Ready for Phase 3** (performance + single-pass diff) after these fixes.
